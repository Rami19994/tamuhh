Create a full Arabic RTL (right-to-left) web platform for a volunteer team that helps University of the People students who are unable to submit proof of secondary school completion because of ongoing violence and difficulty accessing documents in Sweida since July.

The platform must be professional, simple, confidential, mobile-friendly, and built for real operational case management, not just as an informational page.

Main goal:
Build a secure case-management website that allows students to request free and confidential help, while reducing exploitation, preventing workflow leakage, and ensuring that each student sees only the exact step they need at their current stage.

Core principles:
- The service is completely free.
- Anyone asking for money does not represent the team.
- The platform must protect student privacy.
- Students must not see the full workflow.
- Students must not see counselor/advisor email addresses or sensitive contact details except at the final stage and only if their case is approved.
- The platform must follow the principle of “minimum necessary disclosure.”
- The team does not send emails on behalf of students. The student sends from their own email after receiving a customized draft and final instructions.

Language and design:
- The entire website must be in Arabic.
- Full RTL support.
- Clean, modern, trustworthy design.
- Colors: white, dark blue, gray, with subtle green accents.
- Clear, readable typography.
- Mobile-first responsive design.
- Tone: respectful, serious, humanitarian, non-political, and non-commercial.

Public pages:
1) Home page
- Brief explanation of the service.
- Emphasize that it is free, confidential, and organized.
- A clear warning banner: anyone asking for money does not represent us.
- Main CTA button: “تقديم طلب مساعدة”
- Main CTA button: “التحقق من حالة الطلب”

2) “How it works” page
- Show only high-level steps, without revealing sensitive operational details:
  - Submit request
  - Review and verification
  - Customized guidance
  - Follow-up
- Do not reveal internal operational details, counselor names, or escalation structure.

3) Privacy and confidentiality page
- Explain that the platform collects only the minimum necessary data.
- Explain that data is accessible only to authorized personnel.
- Explain that successful case details and sensitive communications are not publicly shared.

4) Exploitation reporting page
- A simple form for reporting anyone asking for money in exchange for help.
- Fields:
  - Name or alias
  - Contact method
  - Upload image or screenshot
  - Additional notes
- Reports must be marked as confidential.

Student request flow:
Create a secure assistance request form with the following fields:
- Full name
- Email used with the university
- Student ID (optional)
- Short case description
- Can you currently access your secondary school certificate? (Yes / No / Partially)
- Region / area as a dropdown list with these options only:
  - Sweida
  - Rural Sweida
  - Jabal al-Sheikh
  - Jaramana
  - Sahnaya
  - Ashrafiyat Sahnaya
- Upload supporting documents if available
- Dedicated field to upload an ID image
- Dedicated field to upload a screenshot of grades/marks
- Confirmation checkbox with the following text:
  “I understand that this service is free and confidential, and I agree not to share any customized instructions or sensitive contact details provided to me.”

After submission:
- Automatically generate a unique case number such as:
  CASE-2026-0001
- Show a confirmation page with the case number
- Send a secure confirmation email to the email address entered in the email field
- Allow the student to later track the case using the case number + email verification code

Case statuses:
- Received
- Under review
- Need more information
- Approved for guidance
- Draft prepared
- Awaiting student sending
- Sent by student
- Follow-up in progress
- Completed
- Closed
- Rejected

Email notification requirements:
- The system must send automatic email notifications to the exact email address entered by the student in the email field
- Email notifications must be triggered whenever the case status changes
- Each email must clearly include:
  - Case number
  - Updated case status
  - Short explanation of what the new status means
  - Any action required from the student, if applicable
  - A secure link or instruction to check the case status page
- The system must send different email content depending on the status value
- Example email updates:
  - Received: confirm that the request has been received
  - Under review: inform the student that the case is currently being reviewed
  - Need more information: explain exactly what information or files are missing
  - Approved for guidance: inform the student that the case has been accepted for the next step
  - Draft prepared: notify the student that a customized email draft is available
  - Awaiting student sending: ask the student to send the prepared email and upload a screenshot after sending
  - Sent by student: confirm that the student submission has been logged
  - Follow-up in progress: inform the student that follow-up is ongoing
  - Completed: inform the student that the case has been completed
  - Rejected: inform the student that the case has been rejected, with a short reason if allowed by admin settings
  - Closed: inform the student that the case is closed
- Email notifications must be in Arabic
- Emails must have clean Arabic RTL formatting
- Emails must be professional, concise, and easy to understand
- Admin users must be able to control whether a student-facing email notification is sent when changing a status
- The system should support default email templates for each status
- The system should allow admins to optionally customize the outgoing student-facing message before sending
- Log every sent email notification in the admin activity log, including timestamp, recipient, case number, status, and whether delivery was attempted successfully

Student portal:
Create a secure student portal that allows the student to:
- Enter the case number and email verification code
- View only the current case status
- Upload requested documents
- Read only the current instruction related to their case
- Receive a customized email draft if approved
- Confirm that they have sent the email
- Upload a screenshot from the “Sent” folder
- Not see any internal notes
- Not see any other cases
- Not see the full workflow map
- Not see the counselor/advisor directory or template library

Very important restriction:
The student must see only one step at a time.
Examples:
- If the case is under review, they only see that it is under review.
- If more information is needed, they only see exactly what is missing.
- If approved, they only see their customized email draft and current sending instructions.
- Do not expose future steps or the complete workflow.

Internal admin dashboard:
Create a secure internal admin panel for the team with role-based access.

Roles:
1) Intake officer
2) Verification officer
3) Drafting and follow-up officer
4) Records officer
5) Super admin

Admin dashboard features:
- View all cases
- Filter by case status
- Search by case number, name, or email
- Add internal notes
- Assign a case to a team member
- Request more information from the student
- Change case status
- Manage private internal templates
- Generate customized email drafts for students
- Log when and to whom sensitive sending instructions were revealed
- Flag suspicious or abusive submissions
- Export brief reports
- Internal activity log
- A clear option for admins to approve or reject a case from the admin page
- If a case is rejected, the admin must be able to add a rejection reason as an internal note or a short student-facing message
- If a case is approved, it should move automatically or manually to the next stage depending on system design
- Authorized admins must be able to access all files and attachments for each case, including:
  - ID image
  - Screenshot of grades/marks
  - Other supporting documents
  - Follow-up screenshots
- In the admin case page, all student-uploaded attachments must be displayed in a clear dedicated attachments section
- The admin must be able to either preview the attached files directly inside the admin page or download them securely for review
- Images should support inline preview inside the admin page and also provide a secure download button
- PDFs should support embedded viewing inside the admin page and also provide a secure download button
- Other file types should display file name, file type, upload date, file size, and provide secure download and open options when supported
- Each attachment should be clearly labeled by category, such as:
  - ID image
  - Grades screenshot
  - Supporting document
  - Sent-email screenshot
- The attachments section in the admin panel should support thumbnail previews for image files
- Attachments should be organized neatly and be easy to review from the admin panel
- Access to viewing or downloading attachments must remain limited to authorized admin roles only
- When an admin changes the case status, the admin page should include an option to send the corresponding email update immediately to the student’s email address
- The admin should be able to preview the outgoing email before sending it
- The admin should be able to edit the short student-facing status message before sending, while keeping the case status value structured internally

Confidentiality and security controls:
- Role-based permissions
- Sensitive fields hidden unless the role allows access
- Store counselor/advisor email addresses and sensitive contact details in a protected internal area only
- One-time reveal option for some sensitive details
- Audit log for every sensitive action
- Two-factor authentication for admin users
- Automatic session timeout
- Rate limiting and bot protection
- Prevent indexing of private pages by search engines
- Secure file upload handling
- Encrypt sensitive data in transit and at rest

Leak-reduction features:
- Customized drafts must support small controlled wording variations between cases
- The system must log which draft version was given to which case
- The student-facing interface must avoid exposing copyable operational guides
- Use expiring links or OTP verification for sensitive steps
- Uploaded screenshots must be visible only to authorized roles
- Support optional hidden trace markers or subtle text differences in generated drafts for internal tracking
